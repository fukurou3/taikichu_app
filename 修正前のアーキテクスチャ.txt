  🗄️ データベース分散戦略

  1. AlloyDB for PostgreSQL (メインデータベース)

  責任範囲: 構造化された重要なデータ

  主要テーブル:
  ├── users (ユーザー情報)
  ├── posts (投稿・カウントダウン)
  ├── follows (フォロー関係)
  ├── comments (コメント)
  ├── likes (いいね)
  ├── admin_roles (管理者権限)
  └── moderation_logs (監査ログ)

  特徴:
  - ACID準拠の強整合性
  - 自動カウンター更新（トリガー）
  - 接続プール（5-20接続）
  - 読み取りレプリカ対応

  2. Firestore (リアルタイムデータ)

  責任範囲: リアルタイム性が重要なデータ

  コレクション構造:
  ├── user_timelines (個人タイムライン)
  ├── global_timelines (グローバル・カテゴリ別)
  ├── notifications (通知)
  ├── real_time_activities (リアルタイム活動)
  ├── trending_data (トレンドデータ)
  ├── live_events (ライブイベント)
  ├── user_feed_cache (フィードキャッシュ)
  └── user_engagement (エンゲージメント分析)

  特徴:
  - リアルタイムリスナー
  - オフライン対応
  - 自動スケーリング
  - セキュリティルール

  3. Redis (高速キャッシュ)

  責任範囲: 高速アクセスとファンアウト処理

  キーパターン:
  ├── timeline:{userId} (個人タイムライン)
  ├── counter:{postId} (投稿カウンター)
  ├── followers:{userId} (フォロワーリスト)
  ├── following:{userId} (フォロー中リスト)
  ├── global_timeline (グローバルタイムライン)
  ├── fanout_queue (ファンアウトキュー)
  ├── session:{sessionId} (セッション)
  └── rate_limit:{userId} (レート制限)

  特徴:
  - サブミリ秒応答時間
  - TTL自動管理
  - 分散カウンター
  - ファンアウト処理

  🔄 データフロー

  投稿作成時のフロー

  1. Client → API: 投稿作成リクエスト
  2. AlloyDB: 投稿データ保存
  3. Redis: カウンター初期化
  4. Fanout Service: フォロワーへの配信開始
     ├── Push (≤1,000フォロワー): 即座に全配信
     ├── Hybrid (1,000-10,000):
  アクティブユーザーはPush、その他はPull
     └── Pull (>10,000): オンデマンド生成
  5. Firestore: グローバルタイムライン更新
  6. Analytics: イベント記録

  タイムライン取得時のフロー

  1. Client → API: タイムライン取得リクエスト
  2. Redis Cache: キャッシュ確認 (TTL: 7日)
     ├── Hit: 即座に返却 (1-5ms)
     └── Miss: ↓
  3. Firestore: リアルタイムデータ取得
     ├── Success: キャッシュに保存して返却
     └── Fail: ↓
  4. AlloyDB: フォロー関係から動的生成
  5. Analytics Service: フォールバック

  🚀 ファンアウト戦略

  Push Strategy (プッシュ方式)

  - 対象: フォロワー数 ≤ 1,000人
  - 方法:
  投稿時に全フォロワーのタイムラインに即座に配信
  - 利点: 読み取り時の高速性
  - 欠点: 書き込み時の負荷

  Hybrid Strategy (ハイブリッド方式)

  - 対象: フォロワー数 1,000-10,000人
  - 方法: アクティブユーザーはPush、非アクティブはPull      
  - 判定: 過去7日以内のログイン履歴
  - 利点: 負荷とパフォーマンスのバランス

  Pull Strategy (プル方式)

  - 対象: フォロワー数 > 10,000人（セレブユーザー）
  - 方法: タイムライン取得時に動的生成
  - キャッシュ: 生成結果をRedisに保存
  - 利点: 書き込み時の負荷軽減

  🔧 サービス層アーキテクチャ

  PolyglotDatabaseService

  主要メソッド:
  ├── executeOperation<T>() // データベース振り分け
  ├── getUserById() // ユーザー取得
  ├── createUser() // ユーザー作成
  ├── getPostsByCategory() // カテゴリ別投稿
  ├── getUserTimeline() // タイムライン取得
  └── getPostCounters() // カウンター取得

  DatabaseConnectionManager

  機能:
  ├── 接続プール管理 (5-20接続)
  ├── ヘルスチェック (5分間隔)
  ├── 自動再接続 (指数バックオフ)
  ├── フェイルオーバー制御
  └── メトリクス収集

  FanoutTimelineService

  機能:
  ├── ファンアウト戦略決定
  ├── バッチ処理 (100件ずつ)
  ├── キュー管理
  ├── 進捗追跡
  └── タイムライン生成

  📊 パフォーマンス指標

  レスポンス時間目標

  | 操作       | 目標時間     | データベース
   |
  |----------|----------|-------------------|
  | タイムライン取得 | 1-5ms    | Redis (キャッシュ)        
    |
  | ユーザー情報取得 | 10-50ms  | AlloyDB           |       
  | 投稿作成     | 50-200ms | AlloyDB + Fanout  |
  | リアルタイム通知 | < 100ms  | Firestore         |       
  | トレンド計算   | 50-100ms | Analytics Service |

  キャッシュTTL設定

  | データ種別       | TTL  | 理由       |
  |-------------|------|----------|
  | ユーザータイムライン  | 7日   | 頻繁に更新される |      
  | グローバルタイムライン | 1時間  | 高頻度更新    |       
  | カテゴリタイムライン  | 2時間  | 中程度更新    |        
  | 投稿カウンター     | 30日  | 長期保持     |
  | フォロー関係      | 24時間 | 日次更新     |

  🛡️ 信頼性とフォールバック

  フォールバックチェーン

  AlloyDB → Firestore → Analytics Service
  Redis → Firestore → AlloyDB
  Firestore → Redis → Analytics Service
  Analytics → Redis → Firestore

  ヘルスモニタリング

  監視項目:
  ├── 接続状態 (Connected/Error/Connecting)
  ├── レスポンス時間 (P95 < 閾値)
  ├── エラー率 (< 0.1%)
  ├── メモリ使用量 (< 80%)
  └── キャッシュヒット率 (> 95%)

  🔐 セキュリティ

  接続セキュリティ

  - AlloyDB: SSL/TLS、プライベートIP
  - Redis: AUTH認証、VPC内通信
  - Firestore: IAM、セキュリティルール

  データ暗号化

  - 保存時: 全データベースで有効
  - 転送時: TLS 1.2以上
  - フィールドレベル: PII暗号化

  📈 スケーラビリティ

  水平スケーリング

  - AlloyDB: 読み取りレプリカ追加
  - Redis: クラスタリング
  - Firestore: 自動スケーリング
  - Cloud Run: オートスケール

  垂直スケーリング

  - 接続プール: 負荷に応じて調整
  - メモリ: Redis最適化
  - CPU: マルチコア活用

  🔍 モニタリング

  メトリクス収集

  収集データ:
  ├── クエリ実行時間
  ├── 接続プール使用率
  ├── キャッシュヒット率
  ├── エラー発生率
  ├── スループット
  └── リソース使用量

  アラート設定

  - データベース遅延 > 200ms (5分間継続)
  - キャッシュヒット率 < 90% (10分間継続)
  - 接続エラー > 10回/分 (2分間継続)
  - メモリ使用量 > 80% (継続監視)

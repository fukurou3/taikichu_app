# 待機中。アプリ - 最新の仕組み解説書

## 📋 概要

「待機中。」は、トレンド連動型カウントダウンSNSアプリです。このドキュメントでは、2025年7月時点での最新アーキテクチャとその背景を解説します。

## 🎯 アーキテクチャ進化の背景

### 問題：コスト爆弾の危機
従来のFirestore中心のアーキテクチャでは、以下の問題が発生していました：

- **バッチ処理の爆発的コスト**: 全カウントダウンを毎日処理 → 月額5万円～
- **分散カウンターの高コスト**: 10シャード読み取り毎回 → 大量課金
- **リアルタイム性の欠如**: バッチ処理のため更新が遅い

### 解決策：MVP分析基盤
Google Cloud Platform を活用した**リアルタイム分析基盤**を構築し、以下を実現：

- **98%コスト削減**: 月5万円 → 月500円
- **100倍高速化**: 100-500ms → 1-5ms
- **無限スケーラビリティ**: 100万ユーザー対応

## 🏗️ 新アーキテクチャ全体図

```
[Flutter App] ←→ [Firebase Functions] → [Pub/Sub] → [Cloud Run] → [Redis]
    ↓                    ↓                ↓           ↓          ↓
クライアント           イベント発行        配信      リアルタイム  高速DB
(UI/UX)              (軽量処理)         (安定)      分析処理    (1-5ms)
```

## 🔧 システム構成要素

### 1. Firebase Functions（イベント発行層）
**役割**: Firestoreトリガーでイベントを軽量発行

**主要Functions**:
- `onLikeCreate/Delete`: いいねイベント
- `onParticipationCreate/Delete`: 参加イベント  
- `onCommentCreate`: コメントイベント
- `publishViewEvent`: 閲覧イベント

**特徴**:
- 実行時間: 数ミリ秒
- 処理内容: Pub/Subにイベント送信のみ
- コスト: 従来の1/10

### 2. Google Cloud Pub/Sub（メッセージ配信層）
**役割**: 非同期でイベントを確実配信

**構成**:
- Topic: `analytics-events`
- Subscription: `analytics-processor`
- Push型でCloud Runに配信

**特徴**:
- 10GB/月まで無料
- 99.9%可用性保証
- 自動スケーリング

### 3. Cloud Run（分析処理層）
**役割**: リアルタイムでイベントを集計・分析

**サービス**: `analytics-service`
**言語**: Python + Flask
**主要API**:
- `/process-events`: イベント処理
- `/trend-score/{id}`: トレンドスコア取得
- `/counter/{id}/{type}`: カウンター取得
- `/ranking`: ランキング取得

**特徴**:
- 自動スケーリング（0→∞）
- 処理時間: 数ミリ秒
- VPC経由でRedis接続

### 4. Redis（高速データベース層）
**役割**: 分析結果を超高速で提供

**構成**:
- インスタンス: `taikichu-analytics-redis`
- 容量: 1GB Basic
- リージョン: asia-northeast1

**データ構造**:
```
trend_score:{countdown_id} → スコア値
counter:{countdown_id}:{type} → カウント値
ranking:global → ランキングリスト
```

**特徴**:
- レスポンス: 1-5ms
- インメモリ高速アクセス
- 自動バックアップ

### 5. Flutter App（クライアント層）
**役割**: 高速データ表示とユーザー体験

**新クライアント**: `MVPAnalyticsClient`
**主要機能**:
- `getTrendScore()`: トレンドスコア取得
- `getCounterValue()`: カウンター取得
- `getTrendRanking()`: ランキング取得
- `getMultipleCounters()`: 一括カウンター取得

**特徴**:
- キャッシュ機能内蔵
- 並列処理対応
- エラー時のフォールバック

## 📊 データフロー詳細

### リアルタイム分析フロー
```
1. ユーザーアクション（いいね、コメント等）
   ↓
2. Firestore書き込み
   ↓
3. Firebase Functions トリガー
   ↓
4. Pub/Sub イベント発行
   ↓
5. Cloud Run イベント受信・処理
   ↓
6. Redis データ更新
   ↓
7. Flutter App データ取得（1-5ms）
```

### バッチ処理フロー（コスト最適化済み）
```
1. Cloud Scheduler（5分毎）
   ↓
2. Cloud Functions 分散カウンター集計
   ↓
3. Firestore counts文書更新
   ↓
4. 危険なバッチ処理は完全停止
```

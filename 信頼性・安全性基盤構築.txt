信頼性・安全性基盤 構築プロジェクト
1. プロジェクトの目的と基本原則
目的:
ユーザーが安心して楽しめる健全なプラットフォームを維持するため、迅速かつ安全にコンテンツモデレーション（不適切投稿の管理等）を実行できる、スケーラブルな運営体制を構築する。

基本原則:

権限の最小化 (Principle of Least Privilege): 運営者であっても、その役割に必要な最低限の権限しか付与しない。

操作の追跡可能性 (Auditing): 全ての管理者操作は、「誰が、いつ、何を、なぜ」行ったかを必ず記録し、追跡可能にする。

データの保全 (Soft Deletion): コンテンツは物理削除せず、状態を変更することでユーザーから見えなくする（論理削除）。これにより、誤操作からの復旧や、法的な要請への対応を可能にする。

バックエンド主導 (Backend-Driven): 全ての運営操作は、必ずサーバーサイドのAPIを経由して実行する。クライアント（管理画面）がデータベースを直接操作することは絶対に許可しない。

Phase 1: 基盤構築 (バックエンド)
このフェーズでは、運営体制の根幹となるサーバーサイドの仕組みを構築します。

task 0
運営体制構築を機に追加のプロバイダとして、メールパスワードを導入します。

Task 1-1: 役割ベースの権限管理 (RBAC) の導入
内容: Firebase Authenticationのカスタムクレームを利用して、運営者の役割を定義する。

詳細:

superadmin: 全ての権限を持つ最上位管理者（開発者など数名に限定）。

moderator: コンテンツの表示/非表示を管理する権限。ユーザー情報の閲覧は可能だが、個人情報の変更や削除はできない。

Firebase Admin SDKを使い、特定のユーザーUIDに対して admin.auth().setCustomUserClaims(uid, { role: 'moderator' }) のように役割を設定する管理スクリプトを作成する。

Task 1-2: データモデルの拡張
内容: 論理削除と監査に対応するため、Firestoreのデータモデルを拡張する。

対象ファイル: lib/models/countdown.dart, lib/models/comment.dart

詳細:

status (String) フィールドを追加: visible (表示), hidden_by_moderator (運営により非表示), deleted_by_user (ユーザーにより削除) といった状態を持つ。

moderationInfo (Map) フィールドを追加:

moderatorId: 操作した運営者のUID

moderatedAt: 操作日時 (Timestamp)

reason: 操作理由 (例: "利用規約違反のため")

notes: 内部向けのメモ

Task 1-3: 管理者用APIエンドポイントの実装
内容: Cloud Run ( analytics-service/main.py ) に、安全な管理者操作のためのAPIエンドポイントを実装する。

詳細:

認証デコレーターの作成: 全ての管理者用APIは、リクエストヘッダーのIDトークンを検証し、roleがsuperadminまたはmoderatorであることを確認するデコレーターを通過させる。権限がない場合は即時403エラーを返す。

実装するAPI:

POST /admin/contents/moderate: コンテンツ（カウントダウン or コメント）のステータスを変更する。

リクエストボディ: { "contentId": "...", "contentType": "comment", "newStatus": "hidden_by_moderator", "reason": "..." }

処理: 対象コンテンツのstatusとmoderationInfoを更新し、監査ログを記録する。

GET /admin/users/search: ユーザーを検索する。

GET /admin/contents/reported: 通報されたコンテンツ一覧を取得する。

Task 1-4: 監査ログの完全な記録
内容: 全ての管理者操作を記録するための専用コレクションを作成し、APIに記録処理を組み込む。

詳細:

Firestoreに moderation_logs コレクションを作成する。

Task 1-3で実装した管理者用APIは、処理が成功した場合、必ずこのコレクションに以下の情報を含むドキュメントを作成する。

moderatorId, targetContentId, targetContentType, action (例: status_change), details (変更前後のデータ), timestamp

Phase 2: 管理画面の実装 (フロントエンド)
このフェーズでは、運営者が直感的に操作できるWebベースの管理画面を構築します。

Task 2-1: 管理画面プロジェクトのセットアップ
内容: メインアプリとは別の、Flutter Webプロジェクトを新規に作成する。

理由: メインアプリのコードベースから完全に分離することで、セキュリティと開発の独立性を保つ。

Task 2-2: 認証とアクセス制御
内容: 管理画面にFirebase Authenticationを用いたログイン機能を実装する。

詳細:

ログイン後、IDトークンからカスタムクレームを読み取り、role (superadmin or moderator) を確認する。

権限がないユーザーはログインページにリダイレクトする。

取得したIDトークンは、以降の全てのAPIリクエストのAuthorizationヘッダーに含める。

Task 2-3: コア機能UIの実装
内容: ユーザー検索、コンテンツ検索、通報キューなど、基本的な管理画面のUIを実装する。

詳細:

ユーザー検索画面: ユーザーIDや名前で検索し、プロフィール情報や投稿履歴を表示する。

コンテンツ管理画面:

通報があったコンテンツを一覧表示するキュー画面。

コンテンツIDやキーワードで投稿を検索できる機能。

各コンテンツに対して「非表示にする」ボタンを設置する。

Task 2-4: モデレーション操作UIの実装
内容: コンテンツを非表示にする際の、安全で使いやすいUIフローを実装する。

詳細:

運営者が「非表示」ボタンをクリックすると、確認ダイアログが表示される。

確認後、非表示理由を選択式または自由記述で入力するモーダルを表示する（理由の記録は必須）。

「実行」ボタンを押すと、Task 1-3で作成した管理者用API (/admin/contents/moderate) にリクエストを送信する。

成功すれば、UI上の表示を「非表示済み」に更新する。

Phase 3: 運用と拡張
Task 3-1: ユーザーからの通報機能の実装
内容: 一般ユーザーが不適切な投稿を運営に通報できる機能を、モバイルアプリ側に実装する。

詳細:

各コメントやカウントダウンに「通報」ボタンを設置する。

ボタンを押すと、通報理由を選択する画面が表示される。

「送信」すると、Firestoreの reports コレクションに「誰が、どの投稿を、なぜ」通報したかの情報が記録される。

重要: この reports コレクションは、Task 1-3で実装した管理者用API /admin/contents/reported が読み取るデータソースとなる。

Task 3-2: 運営ガイドラインとトレーニング
内容: どのような投稿を非表示にするかの一貫した基準をドキュメント化し、運営チームのトレーニングを実施する。

理由: 属人的な判断を避け、公平で一貫性のあるモデレーションを実現するため。

Task 3-3: 法的文書の更新
内容: 弁護士などの専門家と相談の上、利用規約やプライバシーポリシーに、コンテンツの削除やアカウントの停止に関する条項を明記する。
